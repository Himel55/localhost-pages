01 — the reference content

One string. Six panel sections.

Lines 5–45 are almost entirely a single template-literal HTML string — the user-facing manual for everything the other static modules do. It compiles to a .help-panel dialog containing six <dl> definition lists. No logic, no conditionals — just markup.

This single string is the only place in the entire codebase that answers the question "what can the user actually do?" It must be updated by hand whenever keys.js, peek.js, delete.js, pin.js, edit.js, or search.js gains or loses a shortcut. There is no code generation — the help text is authoritative only because a developer keeps it that way.

02 — lazy build

build() runs at most once.

The DOM node is never created at parse time. build() is called inside open() and guards itself with an early return — so the first call injects the panel; every subsequent call is a no-op.

help.jslines 42 — 55
42  let backdrop = null;
43  const toggleBtn = document.querySelector('.help-toggle');
44
45  function build() {
46    if (backdrop) return;
47    backdrop = document.createElement('div');
48    backdrop.className = 'help-backdrop';
49    backdrop.innerHTML = HTML;
50    document.body.appendChild(backdrop);
51    backdrop.addEventListener('click', (e) => {
52      if (e.target === backdrop) close();
53    });
54    backdrop.querySelector('.help-close').addEventListener('click', close);
55  }
42
backdrop starts as null. Its truthiness is the only state flag the module needs — no built boolean, no separate flag.
46
The guard. If backdrop is already a DOM node, return immediately. This makes build() idempotent — open() can call it unconditionally.
47 — 50
The element is created, the HTML template string is set as innerHTML, and the node is appended to body. After this line backdrop is truthy — the guard will block all future calls.
51 — 54
Two close handlers wired once, at build time. Backdrop-click checks e.target === backdrop so clicking inside the panel does not close it — only a click on the darkened overlay itself does. The × button always closes.
03 — open / close / toggle

Three functions, two CSS classes.

Visibility is CSS-driven — the JS only adds and removes .open on the backdrop and .help-open on body. The stylesheet does the rest: transitions, z-index, pointer-events, and the pressed state of the header button.

help.jslines 57 — 92
57  function isOpen() {
58    return backdrop?.classList.contains('open');
59  }
60
61  function open() {
62    build();
63    backdrop.classList.add('open');
64    document.body.classList.add('help-open');
65    toggleBtn?.setAttribute('aria-pressed', 'true');
66  }
67
68  function close() {
69    backdrop?.classList.remove('open');
70    document.body.classList.remove('help-open');
71    toggleBtn?.setAttribute('aria-pressed', 'false');
72  }
73
74  function toggle() {
75    if (isOpen()) close(); else open();
76  }
77
78  toggleBtn?.addEventListener('click', toggle);
79
80  document.addEventListener('keydown', (e) => {
81    if (e.key === 'Escape' && isOpen()) {
82      e.stopImmediatePropagation();
83      close();
84      return;
85    }
86    if (e.key === '?') {
87      const tag = e.target.tagName;
88      if (tag === 'INPUT' || tag === 'TEXTAREA' || tag === 'SELECT') return;
89      e.preventDefault();
90      toggle();
91    }
92  });
57 — 59
isOpen() uses optional chaining. Before build() has run, backdrop is null — the ?. short-circuits to undefined, which is falsy. No null-check needed in callers.
64 — 65
Two class additions, not one. .open on the backdrop makes the overlay visible. .help-open on body lets the stylesheet highlight the header ? button — a purely visual signal that the panel is active. The aria-pressed attribute carries the same signal to assistive technology.
82
stopImmediatePropagation() — not just stopPropagation(). keys.js also listens for Escape on document; this call prevents that second handler from firing and performing a redundant action while the help overlay is consuming the keystroke.
87 — 88
The typing-guard. ? is a valid character in INPUT, TEXTAREA, and SELECT fields — if focus is in any of those, the keydown falls through unchanged. Only when focus is on the page or a non-text element does ? toggle the overlay.

The IIFE wrapper

The entire module is wrapped in an immediately-invoked function expression — (function () { … })(). HTML, backdrop, and the four functions are private to the closure. Nothing leaks onto window.

The help overlay is documentation that ships inside the product

It is the single canonical answer to "what does this UI do?" — but it is authored by hand and must be kept in sync with keys.js, peek.js, delete.js, pin.js, edit.js, and search.js. If any of those modules adds a shortcut, the HTML string on line 5 is the only thing that needs updating. There is no automated check that it is correct.
