One file, read closely

icons.py
turns a string into a glyph.

An app card needs a picture. The registry only ever has a string — maybe a Lucide name, maybe raw SVG, maybe an emoji, maybe nothing at all. This 51-line file decides what that string means and hands back inline-ready HTML. One function does the work; four ordered branches cover every shape the input can take.

01 — top of the file

Four constants set the stage.

Before any function is defined, the module fixes where icons live, what a valid Lucide name looks like, which icon to fall back to, and where to remember results.

5
escape from the standard library — pulled in for exactly one job: neutralising untrusted text in the emoji branch. Its presence here is a hint that this file handles a trust boundary.
10
LUCIDE_DIR points at the package's bundled icon set — static/lucide/ ships 1703 pre-downloaded Lucide SVG files. Resolving relative to __file__ means it works regardless of the process's working directory.
11
The whitelist. A name is "lucide-shaped" only if it is lowercase letters, digits, and hyphens — starting with a letter or digit. That pattern cannot contain a slash, a dot, or .., so a name that matches can never be used to climb out of LUCIDE_DIR.
12
"box" — the universal fallback. A plain outlined square, used both when the input is empty and when a named icon turns up missing.
13
A module-level dict mapping icon name → SVG text. It persists for the life of the process and is never invalidated — a deliberate choice the next section explains.

Why a regex and not just if "/" not in name?

A whitelist is stronger than a blacklist. Rather than enumerate every dangerous character, line 11 states the only acceptable shape and rejects everything else. The same string then doubles as a filename on line 19 — and because the pattern admits no path separators, that file lookup is provably confined to LUCIDE_DIR.

02 — the loader

_read_lucide()  reads disk once.

A small private helper: given a bare icon name, return the SVG text — or None if no such file exists. Every successful read is memoised.

icons.pylines 16 — 24
16def _read_lucide(name: str) -> str | None:
17    if name in _cache:
18        return _cache[name]
19    path = LUCIDE_DIR / f"{name}.svg"
20    if not path.is_file():
21        return None
22    svg = path.read_text(encoding="utf-8").strip()
23    _cache[name] = svg
24    return svg
17 — 18
Cache first. If this name has been read before, the file system is never touched — the stored string is returned immediately.
19
The name is joined to LUCIDE_DIR with an .svg suffix. Safe only because every caller has already passed the name through _LUCIDE_NAME_RE — the helper itself does not re-validate.
20 — 21
A missing file is not an error. is_file() failing returns a quiet None, leaving the decision of what to do next to the caller.
22
The file is read as UTF-8 and strip()-ped — trailing newlines would otherwise sit inside the inlined markup. The result is SVG ready to drop straight into a page.
23 — 24
The successful read is stored under its name, then returned. Note what is not cached: a miss. Line 21 returns before reaching line 23, so a None result is recomputed every time — see the note below.

Why cache at all?

The index page renders one card per registered app, and many apps share an icon (or the default box). Without the cache, a page with twenty cards could hit the disk twenty times for the same file. With it, each distinct SVG is read once per process and served from memory thereafter. The icon set never changes at runtime, so a never-invalidated cache is correct, not a leak.

Misses are not memoised

Only hits land in _cache; a missing file returns at line 21 before the store on line 23. A repeatedly requested bad name therefore re-stats the disk each time — cheap at localhost scale, and it means dropping a new SVG into the folder is picked up without a restart.

03 — the public function

resolve_icon()  and its docstring.

This is the only name the rest of the codebase imports. registry.py calls it for every app it scans. The docstring states the contract: four branches, tried in order.

36    if not value:
37        value = _FALLBACK_NAME
27
The signature is total: it accepts a str or None and always returns a str. There is no error path — every conceivable input produces displayable HTML.
28 — 35
The docstring is the spec. The four branches below implement it line for line; reading this block first makes the rest of the function predictable.
36 — 37
Branch 1, executed here. not value is true for both None and the empty string "". Rather than return early, the function rewrites value to "box" and lets it fall through — so the empty case is handled by the lucide branch below, with no duplicated file-reading logic.

Branch 1 is a rewrite, not a return

The cleanest part of the design. Instead of return _read_lucide("box") at line 37 — which would mean reading and fallback logic in two places — the empty input is simply relabelled "box". From line 38 onward it is indistinguishable from a request for the box icon by name, and branch 3 does the rest.

Empty → box

Lines 36 — 37, already executed above. None or "" becomes the string "box", so the empty case is served by branch 3 with no special-case code of its own.

2

Raw <svg → verbatim

Lines 39 — 40. If the string already is SVG markup, it is returned unchanged and unescaped. The operator who registered the app supplied this; it is treated as trusted code, not data.

3

Lines 42 — 49. A whitelist-shaped name is loaded from static/lucide/. A miss logs a warning and degrades to box.svg — it never raises.

Line 51. The final catch-all. The value is HTML-escaped and wrapped in a <span class="icon-emoji"> — fit for an emoji character, and safe even if it is not.

39 — 40
Branch 2. A prefix check, not a parse. Any string beginning with the literal <svg is assumed to be a complete, well-formed SVG element and is handed back exactly as received — see the trust-boundary note below.
42 — 43
Branch 3. If the value matches the lucide-name pattern, it is passed to _read_lucide. Because the regex already proved the name is path-safe, this call cannot escape LUCIDE_DIR.
44 — 46
The name was well-formed but no such file exists. A warning is logged — useful when an app declares a typo'd icon — and the function retries with _FALLBACK_NAME, "box". A wrong name degrades to a generic glyph rather than a broken card or an exception.
47 — 48
Defence at the bottom of the well: if even box.svg is missing — a broken install — the function returns an empty .icon-emoji span. It still returns valid HTML; it never lets a packaging fault crash a page render.
49
The normal exit of branch 3: the resolved SVG, whether the requested icon or the box fallback.
51
Branch 4. Reached only when the value is non-empty, is not SVG, and is not a lucide-shaped name — typically a literal emoji like 🚀. It is run through escape() and wrapped in a span, so the browser renders the character itself rather than interpreting it.

The trust boundary  — why branch 2 is raw and branch 4 is escaped

Both branches end up as HTML on a page, but they are treated as opposites. Branch 2's <svg… is returned verbatim: it is code, deliberately authored by whoever registered the app, and escaping it would break the markup. Branch 4's value is treated as data of unknown origin — it is pushed through html.escape() so that <, > and & become entities and nothing in it can inject an element. The choice of which path a string takes is made entirely by the startswith("<svg") test on line 39 — that single check is the boundary between trusted markup and escaped text.

Why an unknown lucide name degrades instead of erroring

resolve_icon is called from registry.py while building the list of app cards for the index page. A raised exception there would take down the entire index over one app's bad icon string. So a missing icon is downgraded to a logged warning plus the box glyph: the operator gets a breadcrumb in the logs, the user still gets a working page.

05 — step back

Four things to take away.

One function, total by construction

resolve_icon accepts str | None and always returns a str. There is no failure mode the caller must handle — every input, including a broken install, resolves to displayable HTML. That is what lets registry.py call it unguarded.

The regex is a security control

_LUCIDE_NAME_RE is not just input tidying. By admitting no slashes or dots, it guarantees that the name-to-filename join in _read_lucide stays inside LUCIDE_DIR — a whitelist that doubles as path-traversal defence.

Trusted code vs. untrusted data

Raw <svg is operator-authored and emitted verbatim; everything in the emoji branch is escaped. The same file both passes through and neutralises markup — and the difference is one startswith check.

Cache hits, log misses, degrade gracefully

Every successful read is memoised so a card-heavy index touches disk once per icon. A missing named icon logs a warning and falls back to box; a missing fallback yields an empty span. Nothing here ever raises.
