pin.js is 56 lines of pure DOM logic — no framework, no server round-trip. It maintains a JSON array of app names in localStorage, stamps a .pinned class onto matching rows, and hands a read API to sort.js so pinned apps always float above the rest.

01 — the pinned set

Four small functions own the state.

load(), save(), isPinned(), and toggle() — no class, no object. The entire data model is a JSON array in one localStorage slot.

pin.jslines 6 — 27
6  const KEY = 'lp:pinned';
7
8  function load() {
9    try {
10      const raw = localStorage.getItem(KEY);
11      const arr = raw ? JSON.parse(raw) : [];
12      return Array.isArray(arr) ? arr : [];
13    } catch { return []; }
14  }
15
16  function save(arr) {
17    localStorage.setItem(KEY, JSON.stringify(arr));
18  }
19
20  function isPinned(name) { return load().includes(name); }
21
22  function toggle(name) {
23    const arr = load();
24    const i = arr.indexOf(name);
25    if (i === -1) arr.push(name); else arr.splice(i, 1);
26    save(arr);
27  }
6
lp:pinned — the lp: namespace prefix keeps this key distinct from any app's own storage.
9 — 13
load() is defensively layered. The outer try/catch handles a corrupted or quota-denied read; the inner Array.isArray guard handles the case where valid JSON was stored but is not an array — both paths return a safe [].
16 — 18
save() is deliberately thin — no validation, no deduplication. toggle() owns the logic; save() just serialises.
22 — 27
toggle() reads current state, finds the index — -1 means absent, so push; any other index means present, so splice it out. One branch, no separate pin/unpin functions.
02 — marking rows

markRows() — one pass, one class.

After any state change, the DOM is brought into sync. Every row is inspected; the .pinned class is added or removed — no partial updates, no diffing.

pin.jslines 29 — 35
29  function markRows() {
30    const set = new Set(load());
31    document.querySelectorAll('.row').forEach((row) => {
32      if (set.has(row.dataset.name)) row.classList.add('pinned');
33      else row.classList.remove('pinned');
34    });
35  }
30
The stored array is converted to a Set once before the loop — Set.has() is O(1) versus Array.includes() which is O(n) per row. Trivial at typical scale, but the right choice.
31 — 34
Every .row element carries a data-name attribute set by the server template. The function adds .pinned for matches and explicitly removes it for non-matches — so a newly-unpinned row is cleaned up in the same pass.

Why a full re-mark after every toggle?

It is simpler and safer than tracking which row changed. The entire operation costs one localStorage read and one DOM query — both are synchronous and near-instantaneous at the scale of a local index page. Correctness beats micro-optimisation here.

03 — wiring

Delegated clicks, bfcache restore, and a public API.

Three final responsibilities: handle user gestures, survive the browser back-cache, and expose helpers so sort.js can read the pinned set when ordering.

pin.jslines 37 — 56
37  document.addEventListener('click', (e) => {
38    const btn = e.target.closest('.pin');
39    if (!btn) return;
40    e.preventDefault();
41    e.stopPropagation();
42    const row = btn.closest('.row');
43    if (!row || !row.dataset.name) return;

37 — 39
Delegated listener on document — a single handler catches clicks on any .pin button, including ones added after page load. closest('.pin') bubbles up from the exact element hit; missing means the click was elsewhere, bail early.
40 — 41
preventDefault() stops any anchor navigation; stopPropagation() prevents the row's own click handler (which navigates to the app) from firing — the pin button is inside the row link.
47 — 48
After marking rows, the sort is immediately re-applied by calling wrap.__setSort() — a function sort.js attaches to the .sort-wrap element. Passing the current sort key means the pinned row jumps to or from the top section without waiting for the next interaction.
51 — 52
markRows() runs once on load to stamp any previously-pinned rows. The pageshow listener re-runs it after a bfcache restore — the browser may replay a cached page without re-executing scripts, so the class state could otherwise be stale.
55
window.__lpPinned exports exactly two helpers — isPinned and load. sort.js calls load() to get the full pinned set when building its sorted order. The double-underscore prefix signals "internal to this project, not a public API."

One class, three consumers

.pinned is the single shared signal. pin.js writes it. sort.js reads it — or reads window.__lpPinned.load() — to build the two-section ordering: pinned rows first, unpinned rows below. index.css reads it too, to highlight the pin button visually. Three consumers, one source of truth — the DOM class is the contract.
