One file, read closely

accents.py
gives every app a colour.

Each card on the index needs an identity hue. This file is the whole colour system: a fixed table of 32 muted pastels, and a tiny resolver that turns an app's meta.json hint — or its name — into one stable hex string. No randomness, no global state, 67 lines.

01 — top of the file

One import. That is the whole dependency list.

No FastAPI, no project siblings — this module is pure data and arithmetic. It depends only on the standard library's hashlib, which makes it trivially testable and import-cheap.

02 — the table

NAMED_ACCENTS  — 32 colours, by name.

A plain dict mapping a colour name to a hex string. It is the single source of truth: both the human-facing names and the raw palette are derived from it.

Why a named table at all?

Pure hashing would still produce stable colours. The names exist so an app author can be deliberate — set "accent": "honey" in meta.json and get exactly that swatch. The table is the menu; the hash is only the fallback.

Every colour sits in the same lightness band

None of the 32 is dark and none is white. That is deliberate: the hash picks blindly, so any pick must look correct on the dark UI. A constrained palette is what lets a random index be safe.

04 — the sequence

PALETTE  — the same colours, indexable.

A dict maps names to colours, but the hash needs to pick by number. This one line re-exposes the table as an ordered, length-known tuple.

accents.pyline 47
47PALETTE: tuple[str, ...] = tuple(NAMED_ACCENTS.values())
47
NAMED_ACCENTS.values() yields the 32 hex strings in insertion order; tuple(…) freezes them into an indexable, immutable sequence.
47
Derived, not duplicated. Because PALETTE is built from NAMED_ACCENTS, the two can never drift apart — edit the dict and the tuple updates with it on the next import.
47
A tuple, not a list, signals intent: this is a fixed lookup table, not a collection anything should mutate. Its len() — 32 — becomes the modulus for the hash.

Names and hashes meet here

The chip indices in section 03 — 00 through 31 — are exactly positions into PALETTE. A hashed app name resolves to one of those numbers; a named app jumps straight to a hex. Two doors, one room.

05 — the arithmetic

_hash_index()  — a name becomes a number.

Two lines turn an arbitrary string into a valid PALETTE index. The leading underscore marks it private — only resolve_accent below calls it.

accents.pylines 50 — 52
50def _hash_index(name: str) -> int:
51    digest = hashlib.sha1(name.encode("utf-8")).hexdigest()
52    return int(digest, 16) % len(PALETTE)
51
name.encode("utf-8") turns the string into bytes — sha1() requires bytes, not text. The result's hexdigest() is a 40-character hexadecimal string.
52
int(digest, 16) parses that hex string as one enormous base-16 integer. % len(PALETTE) folds it into the range 0 … 31 — always a valid index, whatever the name was.
50 — 52
Deterministic. SHA-1 is a pure function: the same name always yields the same digest, hence the same index, hence the same colour. There is no random, no clock, no counter anywhere in this file.

Why hash at all — why not a counter?

A counter ("first app gets colour 0, second gets 1…") would reshuffle every app's colour whenever one is added or removed. Hashing the name makes each app's colour depend only on itself — stable for the life of that app, independent of its neighbours.

SHA-1 is used for spread, not security

Nothing here is cryptographic. SHA-1 is simply a cheap, well-distributed way to scatter names evenly across 32 buckets, so similar names (app-1, app-2) don't all collide on the same hue.

06 — the public function

resolve_accent()  — the one entry point.

This is the only function the rest of the codebase imports. registry.py calls it once per app while building each App record. It never raises and never returns None — there is always a colour.

accents.pylines 55 — 67
55def resolve_accent(value: str | None, name: str) -> str:
56    """Return a hex color (#rrggbb) for this app row.
57
58    Resolution order:
59      1. None / empty                  → hash(name) → PALETTE
60      2. exact match in NAMED_ACCENTS  → mapped hex
61      3. anything else                 → hash fallback
62    """
63    if not value:
64        return PALETTE[_hash_index(name)]
65    if value in NAMED_ACCENTS:
66        return NAMED_ACCENTS[value]
67    return PALETTE[_hash_index(name)]
55
Two arguments. value is the raw accent field from meta.json — possibly None. name is the app's directory name, the stable identifier used as the hash seed.
63 — 64
Step 1. not value is true for both None and the empty string — so an app that never set an accent, or set it to "", gets the hashed auto-pick. This is the common case.
65 — 66
Step 2. If value is an exact key in the table — e.g. "honey" — return its mapped hex directly. The app author got the colour they asked for.
67
Step 3. Anything else — a typo, a stray "#ff0000", a removed colour name — falls through to the same hash path as step 1. The unknown value is silently discarded; the app still gets a sensible colour.
1

None / empty → hash

Line 63 — no accent declared. Hash the name into PALETTE. Stable and automatic, zero configuration required.

2

Known name → mapped hex

Line 65 — an exact key in NAMED_ACCENTS. The author's explicit choice wins; the hash is bypassed entirely.

3

Unknown → hash fallback

Line 67 — anything unrecognised. Identical to step 1. A bad value can never produce a broken page, only the auto-pick.

Note what step 3 does not do

It does not accept a raw hex string. "#a8c8a3" in meta.json is not a key in NAMED_ACCENTS, so it falls to the hash like any other unknown value. The only way to pick a specific colour is by one of the 32 names. The vocabulary is closed on purpose — every accent on the index is guaranteed to come from the vetted palette.

Steps 1 and 3 are the same line for a reason

"No accent" and "bad accent" are treated identically — neither is an error worth surfacing. A colour is decorative; a missing or mistyped one should degrade quietly, not break a card or throw mid-scan. So resolve_accent has no error path at all.

07 — where it plugs in

One call, inside the registry.

This module produces nothing on its own. Its output lives on the App record that registry.py builds for every scanned symlink.

The resolved hex is stored as App.accent. The index template then drops it into each card as a CSS custom property — the same mechanism every page of this explainer site uses for --accent.

Broken apps skip this path entirely

When registry.py builds a record for an app it could not load, it hard-codes accent="#d6928f" — a dusty rose — rather than calling resolve_accent. So a consistent off-palette colour flags failure at a glance, and resolve_accent only ever sees healthy apps.

08 — step back

Four things to take away.

Determinism is the whole point

Same app, same colour — on every reload, every restart, every machine. The hash is a pure function of the name, so a card never flickers between hues and the page looks settled the instant it loads.

A constrained palette makes blind picks safe

All 32 colours are desaturated and sit in one lightness band. Because every entry is known to work on the dark UI, the hash can pick one sight-unseen and never produce something unreadable.

It fails quietly, by design

An unknown or mistyped accent is not an error — it falls through to the same hash as no accent at all. resolve_accent has no exception path; a decorative field should never be able to break a scan.

One source of truth, derived twice

NAMED_ACCENTS is the table; PALETTE is just its values, and _hash_index just its length. Edit the dict and the named lookup, the indexable sequence, and the modulus all stay in sync automatically.
