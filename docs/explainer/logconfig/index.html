logconfig.py
keeps the lights always on.

A newly added module whose single job is to make logging dependable. Logs are always on — no environment variable required — and they land in a rotating file at a platform-standard place plus, when you are at a terminal, stderr. The idea: a user who hits a bug already has a file to send.

01 — top of the file

Stdlib only, plus the package version.

The imports declare the whole toolkit: read the platform, read the environment, touch the filesystem, and rotate a file. Nothing here is a third-party dependency.

logging and RotatingFileHandler do the work; platform reads the OS, version, and python release for the banner; os reads the environment for the override and the Windows/Linux base directories; sys exposes stderr; and Path turns everything into real filesystem objects.

The one sibling import. __version__ is read straight from the package so the banner reports the right version even when the project is not pip-installed.
19 — 21
Module-level constants. One shared LOG_FORMAT, then the rotation policy: _MAX_BYTES is 1 MB per file and _BACKUPS is 5 — so the rotating handler keeps roughly 5 MB of recent history before discarding the oldest.

Why read __version__ directly

The obvious alternative, importlib.metadata.version(...), only works when the distribution is installed. Reading the package's own __version__ attribute needs no metadata and no extra dependency — it is correct whether the code was pip-installed or run straight from a checkout.

02 — one piece of state

Where the log path is remembered.

The module holds exactly one mutable value: the resolved log path, so other parts of the app can ask where are the logs? after setup has run.

logconfig.pylines 23 — 25
23# Resolved at setup_logging() time so the /__diagnostics endpoint and the
24# startup banner can report exactly where logs are written.
25_log_path: Path | None = None

It starts as None and is filled in the moment setup_logging() resolves the path. Two readers depend on it: the /__diagnostics endpoint in server.py, which surfaces the path to a user triaging a bug, and the startup banner, which logs it as the very first thing. Storing it here means there is a single source of truth that does not require re-deriving the path.

03 — picking a place

def default_log_path()  a path per platform.

Every operating system has a conventional home for logs. This function returns the right one, so files land where a user (and their OS tooling) would expect to find them.

logconfig.pylines 34 — 43
34def default_log_path() -> Path:
35    """Platform-standard log location."""
36    system = platform.system()
37    if system == "Darwin":
38        base = Path.home() / "Library" / "Logs"
39    elif system == "Windows":
40        base = Path(os.environ.get("LOCALAPPDATA", Path.home())) / "localhost-pages" / "Logs"
41    else:  # Linux / other POSIX
42        base = Path(os.environ.get("XDG_STATE_HOME", Path.home() / ".local" / "state")) / "localhost-pages"
43    return base / "localhost-pages.log"
37 — 38
macOS. platform.system() returns "Darwin", and logs go to ~/Library/Logs — exactly where Console.app and the rest of the system keep theirs.
39 — 40
Windows. The base is %LOCALAPPDATA%\localhost-pages\Logs, falling back to the home directory if LOCALAPPDATA is somehow unset.
41 — 42
Linux / other POSIX. Honours the XDG spec: $XDG_STATE_HOME/localhost-pages, or ~/.local/state/localhost-pages when that variable is unset.
43
Whichever base was chosen, the file is always named localhost-pages.log inside it.

The override still exists

This function is the default. setup_logging() only calls it when no explicit path is given — and __main__.py passes through LOCALHOST_PAGES_LOG when it is set. So the platform path is what you get out of the box, and the environment variable is the escape hatch when you need the file somewhere specific.

04 — the heart of the file

def setup_logging()  wire up the root logger.

This is the function the entrypoint calls. It resolves a path, makes sure the directory exists, and attaches handlers — and it is safe to call more than once.

logconfig.pylines 46 — 68
46def setup_logging(log_file: str | os.PathLike[str] | None = None) -> Path:
52    global _log_path
53    path = Path(log_file) if log_file else default_log_path()
54    path.parent.mkdir(parents=True, exist_ok=True)
55
56    root = logging.getLogger()
57    root.setLevel(logging.INFO)
58    for handler in list(root.handlers):
59        root.removeHandler(handler)
60        handler.close()
61
62    fmt = logging.Formatter(LOG_FORMAT)
63
64    file_handler = RotatingFileHandler(
65        path, maxBytes=_MAX_BYTES, backupCount=_BACKUPS, encoding="utf-8"
66    )
67    file_handler.setFormatter(fmt)
68    root.addHandler(file_handler)
53 — 54
If a log_file was passed it wins; otherwise default_log_path() chooses one. Either way the parent directory is created (parents=True, exist_ok=True), so pointing at a not-yet-existing location never fails.
56 — 60
Idempotent. The root logger is set to INFO, then every existing handler is removed and closed first. Calling setup_logging() twice — as the test suite does — will not stack duplicate handlers or leak open files.
62 — 68
A single Formatter is built from LOG_FORMAT and shared. Then the RotatingFileHandler is created with the 1 MB / 5-backup policy and UTF-8 encoding, given the formatter, and attached. This is the canonical sink — it is always present.

The resolved path is stored in the module-level _log_path (so the diagnostics endpoint and banner can find it) and also returned, so the caller learns the path immediately.

Why stderr is gated on isatty()

Under a daemon manager like launchd or systemd, the process's stderr is itself redirected to a file. If a stderr StreamHandler were always attached, every line would be written twice into that file — once by the rotating handler and once through redirected stderr. The isatty() check means that in the background the file handler alone is the sink, with no duplication, while interactive runs still echo to the console.

05 — reporting out

The path getter, and the startup banner.

Two small functions expose what setup recorded: one returns the path, the other writes the environment summary you reach for first when a report comes in.

logconfig.pylines 83 — 85
83def get_log_path() -> Path | None:
84    """The active log file path, or None if setup_logging() hasn't run."""
85    return _log_path

get_log_path() is the public reader of the module's one piece of state. Before setup_logging() runs it returns None; afterwards it returns the resolved Path. This is what server.py's /__diagnostics endpoint calls to tell a user exactly which file to send.

88def log_startup_banner(logger: logging.Logger, *, host, port, symlinks_dir, app_count=None) -> None:

94    logger.info("symlinks dir: %s", symlinks_dir)
95    logger.info("log file: %s", _log_path)
96    if app_count is not None:
97        logger.info("apps registered: %d", app_count)
98    logger.info("=" * 60)
88
Keyword-only arguments (after the *) make every call site spell out host=, port=, and symlinks_dir= — readable at the call in __main__.py. app_count is optional.
90 — 95
Framed top and bottom by a rule of = signs, the banner records version, python and OS, the bound host/port, the symlinks directory, and the log file path itself — the exact facts you ask for first when triaging a bug report.
96 — 97
The app count is logged only when it was supplied, so callers that do not yet know it can omit it without printing a misleading zero.

Where this gets called

__main__.py calls setup_logging() and then log_startup_banner(...) at boot, so the first lines in every log already pin down the environment. Pair that with get_log_path() feeding server.py's /__diagnostics endpoint, and the loop closes: the logs are always on, and they are always discoverable.

06 — step back

Four things to take away.

Always on, no switch to flip

There is no enable logging environment variable. The moment the app boots, setup_logging() attaches a rotating file handler — so a user who hits a bug already has a log file, with no restart and nothing to turn on first.

Bounded by design

The 1 MB × 5 rotation caps the footprint at roughly 5 MB of history. Logs can run forever without ever filling a disk, and the oldest lines fall off the back automatically.

One sink, or two, depending on context

The file handler is the canonical sink everywhere. The stderr handler is conditional on isatty() — present when a human is watching, absent under a daemon — which quietly avoids the double-logging trap when stderr is redirected to a file.

Discoverable, not just written

Recording the resolved path in _log_path lets the banner announce it and the /__diagnostics endpoint surface it. Writing logs is only half the value — telling the user where to find them is the other half.
