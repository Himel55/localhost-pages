delete.js
delete, but twice.

The × button on every index row is not a single click. It is a two-step confirm — arm, then fire — with no modal, no dialog, and no undo needed. The first click turns the row rose-red and relabels the button; the second sends the DELETE request and removes the row. Ninety-four lines.

01 — arm & reset

Two helpers, one dataset flag.

arm() and reset() are the only state-writers. Everything else reads the flag they set.

delete.jslines 1 — 12
1(function () {
2  function reset(row) {
3    delete row.dataset.confirming;
4    const btn = row.querySelector('.del');
5    if (btn) btn.textContent = '×';
6  }
7
8  function arm(row) {
9    row.dataset.confirming = 'true';
10    const btn = row.querySelector('.del');
11    if (btn) btn.textContent = '× CONFIRM';
12  }
3
delete row.dataset.confirming removes the attribute entirely — not sets it to empty. The CSS selector [data-confirming] in index.css matches on presence, so removing it cleanly withdraws the rose-red styling.
9
Setting dataset.confirming = 'true' writes data-confirming="true" onto the DOM element. From this moment the row is visually armed — CSS handles the colour change without any JS style manipulation.
5, 11
Button label is the only other visible signal. × at rest; × CONFIRM armed. Simple, no icons, no extra elements.

CSS does the heavy lifting

index.css contains a rule keyed on .row[data-confirming] that applies the rose-red background. delete.js never touches style directly — it only manages the dataset flag, and CSS responds.

02 — doDelete

One fetch. Two outcomes.

Success is silent — the row disappears. Failure is loud — an alert and a reset. Either way the UI stays coherent.

delete.jslines 32 — 53
32  async function doDelete(row) {
33    const name = row.dataset.name;
34    try {
35      const res = await fetch('/__apps/' + encodeURIComponent(name), { method: 'DELETE' });
36      if (res.ok) {
37        purgeStorage(name);
38        row.remove();
39      } else {
40        if (window.lpReport) {
41          window.lpReport('delete failed for ' + name + ': ' + res.status + ' ' + res.statusText);
42        }
43        alert('Delete failed: ' + res.status + ' ' + res.statusText);
44        reset(row);
45      }
46    } catch (err) {
47      if (window.lpReport) {
48        window.lpReport('delete request error for ' + name + ': ' + err, err);
49      }
50      alert('Delete failed: ' + err);
51      reset(row);
52    }
53  }
33
row.dataset.name is the app name written into the HTML by the server when rendering the index. It drives the URL — no DOM scraping, no inner-text parsing.
35
encodeURIComponent(name) ensures names with unusual characters survive as a URL segment. The request lands at DELETE /__apps/{name} — the route documented on the server.py page, which validates and removes the symlink.
38
On res.ok — status 204 from the server — the row is removed from the DOM immediately. No confirmation needed; the user already confirmed on the second click.
43 — 51
HTTP error or network failure both call reset(row) after alerting. The row returns to its normal state so the user can try again — no stuck "armed" UI.
40 — 49
Before each alert, both error paths also call window.lpReport(…) — the browser-error logger defined in lp-log.js, which POSTs to the server's /__log endpoint. A failed delete is captured in the diagnostics log a user can send, not just flashed in an alert.
03 — the click handler

Event delegation. One listener, four branches.

A single click listener on document handles every interaction — the delete button, the confirming row, and anywhere else on the page.

delete.jslines 30 — 59
30  document.addEventListener('click', (e) => {
31    const btn = e.target.closest('.del');
32    if (btn) {
33      e.preventDefault();
34      e.stopPropagation();
35      const row = btn.closest('.row');
36      if (!row) return;
37      // Cancel any other rows that were armed.
38      document.querySelectorAll('.row[data-confirming]').forEach((r) => {
39        if (r !== row) reset(r);
40      });
41      if (row.dataset.confirming) {
42        doDelete(row);
43      } else {
44        arm(row);
45      }
46      return;
47    }
50    const confirmingRow = e.target.closest('.row[data-confirming]');
51    if (confirmingRow) {
52      e.preventDefault();
53      e.stopImmediatePropagation();
54      reset(confirmingRow);
55      return;
56    }
57    // Click anywhere else cancels all armed rows.
58    document.querySelectorAll('.row[data-confirming]').forEach(reset);
59  });
33 — 34
preventDefault stops any default button action; stopPropagation ensures the row's own click-to-preview listener never fires when the × button is the target. The two calls are load-bearing — without them a delete click would also open the preview.
38 — 40
Only one row may be armed at a time. Clicking a different row's × resets all currently armed rows first. Clean, predictable UI — no two rows glowing red at once.
41 — 44
The confirm gate. If data-confirming is already set — this is the second click — call doDelete. Otherwise arm the row. Two clicks required; the logic is explicit.
50 — 55
A click anywhere inside an armed row (but not on the × button) cancels the confirm and does not navigate. stopImmediatePropagation here — stronger than stopPropagation — prevents other listeners on document from treating this click as a row-open event.
58
The fallback branch — a click anywhere outside any armed row resets all armed rows silently. No alert, no disruption; armed state just dissolves.
04 — Esc handling

Cooperative Esc — cancel only when needed.

A keydown listener cancels armed rows on Escape. The subtle part: it only blocks the event when it actually has work to do.

delete.jslines 64 — 70
64  document.addEventListener('keydown', (e) => {
65    if (e.key !== 'Escape') return;
66    const armed = document.querySelectorAll('.row[data-confirming]');
67    if (!armed.length) return;
68    e.stopImmediatePropagation();
69    armed.forEach(reset);
70  });
67
Early exit when nothing is armed. If no rows hold data-confirming, delete.js returns immediately without touching the event — other Esc listeners are free to handle it normally.
68
stopImmediatePropagation fires only when there is actually something to cancel. It prevents keys.js's own Esc handler from running in the same frame — cancelling a delete confirm and clearing keyboard focus would be competing actions.

The cooperative-Esc pattern

delete.js is loaded before keys.js, so its listener runs first. By only calling stopImmediatePropagation() when it actually consumed the event — not unconditionally — it behaves as a polite participant. When no rows are armed, keys.js's Esc handling runs as if delete.js were absent. This is the pattern to follow whenever two independent modules share a key.
