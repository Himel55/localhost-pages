localhost-pages/technical-explanation/templates/index.html live
One file, read closely

index.html
is the one template.

Pure structure, zero JavaScript framework. This single Jinja2 template is rendered by render_index() in index.py, fed a list of App records, and served at GET /. Everything the user sees on the launch page — the brand, the app list, the controls — lives here. The scripts come after; the HTML works without them.

01 — head

One stylesheet. No fonts, no framework.

The entire <head> is seven lines. No Google Fonts, no bundler output, no inline styles — just a single <link> to the package's own stylesheet.

The title is always localhost — static, no Jinja2. There is no per-app title injection here; the browser tab simply names the host.
6
/__static/ is the package's reserved asset namespace, mounted in server.py. The double-underscore prefix ensures no user app can shadow it. index.css is the only dependency.

03 — the apps loop

One {% for %} renders the entire list.

Lines 46 – 72. An empty-state branch; then a Jinja2 for that emits one <li> per App record — icon, number, title, description, controls.

51 — 52
style="--accent: {{ app.accent }}" — each row gets its own CSS custom property, set inline. The accent drives the coloured left-border and icon tint in index.css. No inline colours anywhere else; just this one variable.
53 — 54
data-name and data-mtime are the contract between the server-rendered HTML and the client scripts. delete.js reads data-name to know which symlink to remove; sort.js reads data-mtime to order by modification time.
56
{{ "%02d"|format(loop.index) }} — Jinja2's format filter applied to Jinja's built-in loop.index. Produces zero-padded counters: 01, 02, …
57
{{ app.icon_svg|safe }} — the |safe filter tells Jinja2 not to escape the value. The SVG string is built by icons.py, which sanitises it before handing it to the template; trusting it here is intentional.
59 — 60
When app.error is set, the broken class is added to the <li>, a BROKEN badge appears in the title, and the description slot is replaced by the error message itself. Three changes, one condition.

The list is an <ol> — ordered, semantic. The visible two-digit numbers are decorative; the browser's own list count is the structural truth. pin.js and sort.js reorder DOM nodes but leave the <ol> intact.

Jinja2 placeholders in this file — every dynamic value and what it renders:

{{ count }}
Total number of registered apps — from len(apps) in index.py
{{ broken }}
Count of apps where app.error is set
{{ app.accent }}
A CSS colour string, e.g. #bdd0b8 — set as a CSS custom property on the row
{{ app.name }}
The symlink filename — used in the href and the delete button's aria-label
{{ app.mtime_iso }}
ISO-8601 modification timestamp on the symlink — read by sort.js for "recently updated" order
{{ "%02d"|format(loop.index) }}
Zero-padded loop counter — the visible two-digit number on each row
{{ app.icon_svg|safe }}
Pre-built SVG string from icons.py — rendered unescaped into the icon slot
{{ app.title }}
Display name — the app directory name, title-cased
{{ app.description }}
One-line description read from the app's metadata, or a default
{{ app.error }}
Error message string when the app's target is missing or broken; absent when healthy
04 — script tags

Eleven scripts. Load order is execution order.

reload.js loads first and is marked with a sentinel attribute. lp-log.js is the first of the UI scripts so it can capture errors thrown by every module after it. The remaining modules load in DOM order — and several expose hooks that later ones consume.

data-localhost-pages-reload is the sentinel. inject.py searches for this attribute before splicing the reload script into any served HTML — so the index page is never double-injected. data-app="__index" tells reload.js to subscribe to the INDEX_TOPIC SSE channel.
74 — 82
Nine interaction modules, each owning one feature. visited.js records clicks before sort.js loads — so "recently visited" sort is available on first render. keys.js loads after edit.js, peek.js, and search.js so it can bind keyboard shortcuts to functions those modules expose.
reload.js
Opens the SSE connection; calls location.reload() on a change event
lp-log.js
Captures browser errors and POSTs them to /__log; loads first so it sees failures in the modules below
delete.js
Sends DELETE /__apps/{name} on ✕ click; removes the row on 204
visited.js
Persists click timestamps in localStorage for visit-based sorting
pin.js
Persists pinned apps; reorders the <ol> to float them to the top
sort.js
Wires the sort dropdown; reorders rows by alpha, mtime, visit frequency, or manual drag
edit.js
Toggles edit mode — shows delete and drag handles, enables drag-to-reorder
keys.js
Global keyboard shortcuts — / focuses search, ? opens help, etc.
peek.js
Preview modal — hovering or selecting an app shows a live iframe thumbnail
search.js
Filters rows in real time by data-name against the search input
help.js
Builds and toggles the keyboard-shortcut overlay triggered by the ? button
05 — step back

Three things to take away.

Progressive enhancement — the page works with zero JS

The app list is plain <ol>/<li> with real <a href> links. Every control is a native <button>. The scripts enhance — they do not build the page. Disable JavaScript and the list still renders and links still work.

The data-* attributes are the server–client contract

data-name and data-mtime on each row are how the server communicates app identity and freshness to the scripts — no JSON endpoint, no hydration, no framework. The contract is readable in View Source.

Every dynamic value is a Jinja2 placeholder — nothing else

The template contains no Python logic, no filters beyond |safe and |format, and no macros. index.py does all the work before handing data to the template. The template just interpolates — exactly as a template should.
