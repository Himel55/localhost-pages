01 — the whole file

Nothing is omitted here.

The code block below is not an excerpt. It is __init__.py in its entirety — every byte of it.

__init__.pyline 1 — 1
1__version__ = "0.1.0"

One module-level name, __version__, bound to the string "0.1.0". There are no imports, no functions, no classes, and no code that runs for effect. Importing the localhost_pages package executes this single assignment and nothing else.

The name __version__ is a long-standing Python convention — the place a package records its own release version so callers can read localhost_pages.__version__ at runtime.

02 — the concept

What __init__.py is for.

It is not an ordinary module. Python treats a file with this exact name specially.

It marks a directory as a package. When __init__.py sits inside src/localhost_pages/, Python recognises that directory as the importable package localhost_pages — and the sibling files (server.py, watcher.py, and the rest) become its submodules, importable as localhost_pages.server and so on.

It runs once, on first import. The first time anything imports the package — or any module inside it — Python executes __init__.py top to bottom and caches the result. Whatever names it defines become attributes of the package object itself.

It is the conventional home for package-level metadata. Because it runs first and its names attach to the package, __init__.py is where small package-wide facts live — most commonly __version__. It can also re-export a curated public API, but it is under no obligation to do either.

This package's __init__.py does exactly one of those jobs

It marks the directory as a package (just by existing), and it defines one metadata name. It does not re-export anything — so the choice of what is "public" is left to each submodule and to how callers import them directly.

03 — by design

Why this one holds no logic.

A one-line __init__.py is not laziness. In a well-factored package it is the goal.

The real code lives in siblings

Routing, file watching, pub/sub, scanning, rendering — each has its own module beside this file. Keeping __init__.py minimal means there is no ambiguity about where any given behaviour is implemented.

Importing the package is cheap and side-effect-free

With no imports and no executable statements beyond one assignment, importing localhost_pages does no work, touches no filesystem, and starts no servers. A heavy __init__.py would impose that cost on every importer, including the test suite.

No re-exports means no import cycles

If __init__.py imported its own submodules to re-export their names, every one of those submodules would load whenever the package is touched — and any submodule importing the package back would risk a circular import. An empty-of-logic init sidesteps that entirely.

This file is the single source of the version

pyproject.toml does not repeat the version. It declares dynamic = ["version"] and points [tool.hatch.version] at this file's __version__ — so the packaging version is derived from this one line at build time. There is exactly one place to edit, and the two can never drift apart.

04 — step back

Three things to take away.

A file's presence can be its main job

Most of what __init__.py does for this package is simply existing — that alone is what makes src/localhost_pages/ an importable package.

Small is a feature, not a gap

A one-line init keeps imports fast, side-effect-free, and cycle-free. There is nothing missing here — the brevity is the design.

One fact, in one place

The release version lives here as __version__ and nowhere else. pyproject.toml reads it through Hatch's [tool.hatch.version], and the running app reads it through logconfig.app_version() — so a single edit to this line updates the package metadata, the startup banner, and the diagnostics bundle at once.
