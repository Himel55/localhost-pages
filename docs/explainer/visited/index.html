One file, read closely

visited.js
is the badge.

A self-contained IIFE that compares each index row's file mtime against the last time you clicked into it. When the file is newer, a coloured left bar appears. When you click, the timestamp is written to localStorage and the bar disappears. Sixty-one lines — no framework, no imports, just storage and DOM.

01 — read & mark

Parse the timestamp. Toggle the badge.

lastVisitedAt reads one localStorage entry and parses it back to a Unix millisecond integer. markRows runs that check against every row's data-mtime and adds or removes .has-update.

5  const STORAGE_PREFIX = 'lp:visited:';
6
7  function lastVisitedAt(name) {
8    const v = localStorage.getItem(STORAGE_PREFIX + name);
9    return v ? Date.parse(v) : 0;
10  }
11
12  function markRows() {
13    document.querySelectorAll('.row').forEach((row) => {
14      const name = row.dataset.name;
15      const mtimeIso = row.dataset.mtime;
16      if (!name || !mtimeIso) return;
17      const mtime = Date.parse(mtimeIso);
18      if (Number.isNaN(mtime)) return;
19      if (mtime > lastVisitedAt(name)) {
20        row.classList.add('has-update');
21      } else {
22        row.classList.remove('has-update');
23      }
24    });
25  }
5
STORAGE_PREFIX namespaces every key as lp:visited:<name> — a simple convention that keeps visited.js's keys distinguishable from the visit-count keys used later.
9
Absent entry returns 0 — epoch. Any real file mtime will be greater, so an app that has never been clicked is always flagged as updated on first view.
14 — 15
Both values come from data attributes stamped on each row by the server's index template — data-name and data-mtime (an ISO 8601 string).
17 — 18
Date.parse returns NaN on a malformed string. The explicit isNaN guard means a bad mtime is silently skipped — the row is neither badged nor cleared.
19 — 23
The comparison is a strict greater-than. Equal timestamps — visited at the exact same millisecond as the mtime — do not trigger the badge. Both branches are explicit: the else actively removes the class so a row already badged from a previous render is cleared if the user visited in the meantime.
02 — record a visit

Stamp now. Bump the counter. Re-sort.

recordVisit writes two localStorage entries and, if the active sort depends on visit data, immediately re-applies it so the row jumps to its new position without waiting for a page reload.

visited.jslines 27 — 47
27  function recordVisit(row) {
28    if (!row || !row.dataset.name) return;
29    const name = row.dataset.name;
30    // Always stamp with "now" so recently-visited sorts by actual click
31    // time. The has-update indicator still works because now > any past
32    // mtime, and if the file gets touched later (mtime > now) the badge
33    // reappears on next render.
34    const stamp = new Date().toISOString();
35    localStorage.setItem(STORAGE_PREFIX + name, stamp);
36    const countKey = 'lp:visits:' + name;
37    const current = parseInt(localStorage.getItem(countKey) || '0', 10) || 0;
38    localStorage.setItem(countKey, String(current + 1));
39    row.classList.remove('has-update');
40    // If the active sort depends on visit data, re-apply it so the row
41    // jumps to its new position. Otherwise the order would only update
42    // on the next page reload.
43    const mode = localStorage.getItem('lp:sort');
44    if (mode === 'recently-visited' || mode === 'most-visited') {
45      document.querySelector('.sort-wrap')?.__setSort?.(mode);
46    }
47  }
34
new Date().toISOString() — always now, never the file's mtime. This is deliberate: the recently-visited sort orders by real click time, not by when the file was last modified.
36 — 38
A second key, lp:visits:<name>, increments a visit counter. The parseInt(…) || 0 double-default handles both a missing key (null) and a corrupt value (NaN).
39
The badge is cleared immediately on click — before the navigation — so the index looks clean if the user presses back and returns.
43 — 45
The __setSort hook. sort.js exposes this function on the .sort-wrap DOM node so that visited.js can trigger a re-sort without importing sort.js — modules cooperate through the DOM, not through bundler imports.

Why stamp "now" and not the file's mtime?

If the stored timestamp were the file's mtime, two consecutive visits to different apps at the same mtime would be indistinguishable. Storing click time means the recently-visited sort is stable and reflects real browsing order. The badge logic is unaffected: the next time the file is touched, its new mtime will exceed the stored click time and the badge reappears.

03 — wiring

One listener. One export. One event.

The three lines that connect the two functions to the rest of the page — a delegated click handler, a window-level export for peek.js, and a pageshow listener that fires on bfcache restores.

A single delegated click listener on document — not one per row. closest('a.body') walks up from whatever element was actually clicked, so it catches clicks on child elements inside the anchor.
57
window.__lpRecordVisit is the cross-module contract with peek.js. When the preview modal opens an app, it calls this function directly — so a preview counts as a visit and clears the badge without requiring the user to navigate away.
60
pageshow fires on first load and on bfcache restore. DOMContentLoaded alone would miss the back/forward case — the page is served from cache and no DOM event fires, but pageshow always does.
04 — step back

Three things to take away.

localStorage as the only state store

There is no server-side session, no cookie, no API call. The entire "what changed since you looked" feature lives in the browser's own storage — which means it works offline, survives page reloads, and costs exactly zero server resources.

Modules cooperate through window hooks, not imports

visited.js reaches sort.js via .sort-wrap.__setSort and is reached by peek.js via window.__lpRecordVisit. No bundler, no module graph — coordination happens through named properties on DOM nodes and the global object.

pageshow is the right hook for bfcache

Browsers can serve a back-navigation from an in-memory snapshot without firing DOMContentLoaded or load. Listening on pageshow is the only way to guarantee badges are re-evaluated after the user clicks back from an app they just visited.
