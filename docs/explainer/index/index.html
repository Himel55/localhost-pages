01 — top of the file

Imports and the template path.

Four lines of imports name the whole job: a Path to the templates, the three pieces of Jinja2 needed to render them, and the App type this module consumes but never produces.

The three Jinja2 names used below: Environment is the renderer, FileSystemLoader reads templates off disk, and select_autoescape builds the escaping policy.
7
App is imported from registry.py — the typed, frozen dataclass that scan() returns. This module reads its fields; it never constructs one.
9
_TEMPLATES resolves to templates/ beside this file, regardless of the process's working directory. __file__ is the absolute path of index.py, so .parent / "templates" is stable.

Why _TEMPLATES and _env are underscore-prefixed

The leading underscore marks them as module-private. The only name this module exposes is render_index — the environment and its template directory are implementation detail that callers never touch.

02 — module-level state

One Environment, built once.

The Jinja2 environment is created at import time, not inside the function. It is the only piece of state in the file — and the only place security is configured.

index.pylines 10 — 13
10_env = Environment(
11    loader=FileSystemLoader(_TEMPLATES),
12    autoescape=select_autoescape(["html"]),
13)
10
The Environment is the compiled, cached renderer. Assigning it to a module-level name means it is constructed exactly once — the first time index.py is imported.
11
FileSystemLoader points Jinja2 at _TEMPLATES. When get_template("index.html") is called later, the loader reads that directory to find the file and the environment caches the compiled result.
12
select_autoescape(["html"]) turns on autoescaping for templates whose name ends in .html — so every value interpolated into the page is HTML-escaped by default.

Why autoescaping is on

The template renders app-supplied title and description strings — values that ultimately come from a meta.json file inside whatever directory a user symlinked in. Those are untrusted. With autoescaping on, a title containing <script> is emitted as inert text, not executed. It is the one line in this file standing between the registry and a stored-XSS hole.

Why built once at module level, not per request

A Jinja2 Environment compiles and caches templates. Rebuilding it on every call to render_index would re-parse index.html on every page load and throw the cache away each time. Defining _env at module scope means the template is compiled once and reused for the life of the process — the function below just feeds it data.

03 — the only function

render_index() — data into a page.

One signature, three moves: flatten each App into a plain dict, count the broken ones, hand the lot to the template. Called by server.py's GET / route on every hit.

index.pylines 16 — 29
16def render_index(apps: list[App]) -> str:
17    enriched = [
18        {
19            "name": a.name, "title": a.title, "description": a.description,
20            "icon": a.icon, "icon_svg": a.icon_svg, "accent": a.accent,
21            "error": a.error,
22            "mtime_iso": a.last_modified.isoformat() if a.last_modified else "",
23        }
24        for a in apps
25    ]
26    broken = sum(1 for a in apps if a.error)
27    return _env.get_template("index.html").render(
28        apps=enriched, count=len(apps), broken=broken,
29    )
16
Takes a list[App] — the exact return type of registry.scan() — and promises a str of HTML. The whole contract is in the signature.
17 — 25
A list comprehension that flattens each frozen App dataclass into a plain dict. The template receives ordinary dicts, not custom objects — nothing in the template needs to know the App type exists.
19 — 21
Seven fields copied straight across: name, title, description, icon, icon_svg, accent, error. The dict deliberately drops App.target — the resolved filesystem path is server-side detail the page never shows.
22
The one transformed field. App.last_modified is a datetime | None; this turns it into an ISO-8601 string, or "" when the app is broken and has no timestamp. The if … else guards the None case so .isoformat() is never called on nothing.
26
broken counts apps whose error field is set. A generator feeds sum a 1 per failing app — the template uses this for a header summary without having to scan the list itself.
27 — 29
get_template("index.html") fetches the compiled template from the environment's cache, then .render() binds three names into it: apps (the dict list), count (total), and broken. The rendered HTML string is returned.

Why mtime_iso is pre-formatted here

Line 22 does the None-check and the .isoformat() call in Python, before the template ever runs. The template receives a finished string and can interpolate it directly — no date logic, no conditional, no formatting filter in the markup. Keeping that work in index.py leaves index.html a logic-free document of slots.

04 — the shape of it

Find, render, display — three jobs, three files.

The reason this file is only 29 lines is that it does exactly one of those jobs. The other two live elsewhere, and the boundaries between them are sharp.

registry.py
Finds the facts. scan() walks the symlinks directory, reads each meta.json, resolves icons and accents, and returns typed, frozen App objects — including Apps that carry an error.
index.py
Renders the facts. Takes those App objects, converts them to plain dicts, counts the broken ones, and asks Jinja2 for HTML. It performs no filesystem access and makes no decisions about what an app is.
templates/index.html
Displays the facts. A logic-free Jinja2 template that lays the dicts out as cards. Every value it shows arrives pre-computed.

The typed-object → plain-dict handoff

The App dataclass is the right shape for Python code: frozen, typed, attribute access, easy to test. A Jinja2 template wants something simpler. Line 17's comprehension is the deliberate seam between the two worlds — typed objects on the registry side, plain dicts on the template side. That conversion is most of what render_index exists to do.

05 — step back

Four things to take away.

This file finds nothing

There is not one filesystem call inside render_index. It receives a ready-made list[App] and only reshapes it. All discovery — symlinks, meta.json, broken-link detection — happened earlier, in registry.py.

Autoescaping is the security boundary

App titles and descriptions are untrusted, user-supplied strings. select_autoescape(["html"]) on line 12 is what makes interpolating them into the page safe — the single configuration choice that matters most in this file.

The environment is built once

_env lives at module scope, so the template is compiled and cached on first import and reused for every request. render_index stays a thin, stateless function over that shared, warm environment.

The template stays logic-free

Dropping target, pre-formatting mtime_iso, and pre-counting broken all push computation into Python. By the time index.html runs, it has nothing left to decide — just slots to fill.
