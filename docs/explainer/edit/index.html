01 — entering edit mode

One class. CSS handles the rest.

setEditing(on) flips a single class on body — index.css then shows drag handles, hides the ✕ buttons, and disables click-to-preview. JavaScript does only what CSS cannot: flip the button label, update aria-pressed, and set draggable per row.

02 — the drag handlers

Four events. Live DOM reorder.

The reorder happens inside dragover — the DOM is updated on every mouse position, so the row moves with the cursor in real time. dragstart and dragend manage the drag-source marker; dragover does the geometry and moves the node.

edit.jslines 31 — 62
31  // Drag-and-drop reorder. Native HTML5 DnD is enough for in-list moves.
32  let dragging = null;
33
34  list.addEventListener('dragstart', (e) => {
35    if (!document.body.classList.contains('editing')) return;
36    const row = e.target.closest('.row');
37    if (!row) return;
38    dragging = row;
39    row.classList.add('drag-source');
40    e.dataTransfer.effectAllowed = 'move';
41    e.dataTransfer.setData('text/plain', row.dataset.name);
42  });
43
44  list.addEventListener('dragend', (e) => {
45    const row = e.target.closest('.row');
46    if (row) row.classList.remove('drag-source');
47    dragging = null;
48    list.querySelectorAll('.drag-over').forEach((r) => r.classList.remove('drag-over'));
49  });
50
51  list.addEventListener('dragover', (e) => {
52    if (!dragging) return;
53    e.preventDefault();
54    const target = e.target.closest('.row');
55    if (!target || target === dragging) return;
56    const rect = target.getBoundingClientRect();
57    const before = (e.clientY - rect.top) < rect.height / 2;
58    list.querySelectorAll('.drag-over').forEach((r) => r.classList.remove('drag-over'));
59    target.classList.add('drag-over');
60    if (before) target.before(dragging);
61    else        target.after(dragging);
62  });
34 — 41
dragstart — records which row is moving into the module-level dragging variable, marks it with .drag-source (CSS dims it), and stamps effectAllowed = 'move' so the cursor shows the right icon. The setData call is required by Firefox — without it Firefox cancels the drag immediately.
44 — 49
dragend — fires whether the drop succeeded or was cancelled. Clears .drag-source, nulls dragging, and sweeps any stale .drag-over highlights. Clean state guaranteed.
53
e.preventDefault() is what tells the browser a drop is allowed here. Without it, the browser shows the "not-allowed" cursor and the drop event never fires.
56 — 61
The geometry. getBoundingClientRect() gives the target row's screen position. Dividing its height in half creates a top half / bottom half threshold — cursor above the midpoint calls target.before(dragging); below calls target.after(dragging). The DOM reorders live, every pixel of the drag.

Why event delegation on .apps, not per-row listeners?

Rows are dynamically rendered and can be added or removed. Attaching four listeners to the container means newly inserted rows are automatically covered — no re-wiring required.

03 — persistence

Write order. Force manual. Survive reload.

The visual move alone is ephemeral — a page reload would re-run sort.js and restore whatever order was last stored. The drop handler closes that gap: write the new sequence to lp:order, then switch the sort mode to manual so sort.js will replay it on the next render.

edit.jslines 64 — 73
64  list.addEventListener('drop', (e) => {
65    e.preventDefault();
66    list.querySelectorAll('.drag-over').forEach((r) => r.classList.remove('drag-over'));
67    if (!dragging) return;
68    // Persist the new order across the entire list.
69    const order = Array.from(list.querySelectorAll('.row')).map((r) => r.dataset.name);
70    localStorage.setItem(ORDER_KEY, JSON.stringify(order));
71    // Switch to manual sort so the new order sticks on next reload.
72    sortWrap.__setSort?.('manual');
73  });
65 — 66
preventDefault again — required to accept the drop, since the browser defaults to navigating to dropped content. Cleanup sweeps any residual .drag-over highlights that dragend hasn't cleared yet.
69 — 70
Read the current DOM order — not the old array, the live nodes — map each to its data-name attribute, and write the resulting array as JSON to lp:order. Snapshotting the DOM after the move guarantees the stored order matches what the user sees.
72
sortWrap.__setSort?.('manual') — optional chaining because sort.js may not have run yet. Calling it switches the sort dropdown to manual and writes lp:sort = 'manual' to localStorage. On next reload, sort.js reads that key, reads lp:order, and reorders the DOM to match — the hand-arranged sequence persists indefinitely.

Why switch sort mode, not just write the order?

sort.js only consults lp:order when the current sort mode is manual. Writing the array alone would be ignored — the sort mode check comes first. Flipping the mode is what activates the stored sequence.

Native DnD — no library required

HTML5 drag-and-drop covers everything an in-list vertical reorder needs: source tracking, position geometry, live DOM manipulation, and drop confirmation. The platform API is verbose but complete — the 40 lines here replace the entire surface area of a dedicated drag library.
