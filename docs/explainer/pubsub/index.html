01 — what it is for

A topic broker, in-process.

Publish/subscribe in its smallest honest form: producers name a topic, consumers ask for a queue, and a publish drops the topic name into every queue listening on it.

pubsub.pylines 1 — 9
1from __future__ import annotations
2
3import asyncio
4from collections import defaultdict
5
6
7class PubSub:
8    """Async fan-out: publish a topic name; each subscriber to that topic
9    gets the topic name dropped into its asyncio.Queue."""
1
from __future__ import annotations makes every type annotation a lazily-evaluated string. That is what lets line 12 write asyncio.Queue[str] as a subscript without Python evaluating it at class-definition time.
3 — 4
The entire dependency list. asyncio supplies the queue primitive; defaultdict supplies the auto-vivifying topic map. No third-party package — the broker is pure standard library.
8 — 9
The docstring states the contract precisely: the payload is the topic name. A subscriber to "myapp" that receives a message receives the string "myapp" — there is no separate event body.

Why a 26-line broker is enough — no Redis

A real message bus exists to cross process and machine boundaries and to survive restarts. localhost-pages does neither: publisher (the watcher) and subscribers (SSE handlers) live in the same event loop, same process. Messages need only travel between coroutines, and a lost message on restart is irrelevant — the page just reloads. An asyncio.Queue already is a single-process channel; PubSub only adds topic routing on top.

02 — construction

__init__  one field, auto-vivifying.

All the broker's state is a single dictionary mapping topic name to a set of queues. The choice of defaultdict(set) erases every "does this key exist yet?" branch from the methods below.

pubsub.pylines 11 — 12
11    def __init__(self) -> None:
12        self._subs: dict[str, set[asyncio.Queue[str]]] = defaultdict(set)
12
The type reads as the data model: topic name → set of queues. Each queue carries str messages — the topic names from the docstring.
12
A set, not a list: subscribers are unordered and each must appear at most once. It also makes discard in unsubscribe an O(1) operation rather than a linear scan.
12
defaultdict(set) means reading a never-seen topic returns a fresh empty set instead of raising KeyError. subscribe relies on this — self._subs[topic].add(q) works even on the first subscriber to a brand-new topic.

The leading underscore is a deliberate boundary

_subs is private by convention. Callers never reach into the dict; they only ever go through the three methods. That keeps the auto-vivification quirk — reading a topic mutates the dict — contained inside this file, where line 21 is written to account for it.

03 — method one

subscribe  hand out a queue.

A consumer calls this to join a topic. It gets back its own private asyncio.Queue — the receiving end of the channel.

pubsub.pylines 14 — 17
14    def subscribe(self, topic: str) -> asyncio.Queue[str]:
15        q: asyncio.Queue[str] = asyncio.Queue()
16        self._subs[topic].add(q)
17        return q
15
A brand-new, unbounded queue per call. Two subscribers to the same topic each get a distinct queue object — fan-out, not a shared work queue.
16
Register the queue under the topic. If the topic has never been seen, defaultdict creates the empty set first; .add then inserts the queue. No branch, no setdefault.
17
The queue is returned to the caller. From here the broker and the caller hold the same object — the broker writes one end, the caller reads the other.

Where this call comes from

It is server.py's /__events route. When a browser opens the SSE stream, the handler calls pubsub.subscribe(topic) and then loops await q.get() — every message it pulls becomes a change event written down the wire. The returned queue is that tab's live-reload mailbox.

04 — method two

unsubscribe  leave, and clean up.

The mirror of subscribe. It removes one queue from a topic — and, crucially, removes the topic itself once the last subscriber is gone.

pubsub.pylines 19 — 22
19    def unsubscribe(self, topic: str, q: asyncio.Queue[str]) -> None:
20        self._subs[topic].discard(q)
21        if not self._subs[topic]:
22            self._subs.pop(topic, None)
20
discard, not remove: it silently does nothing if the queue is already absent. Calling unsubscribe twice — or with a stale queue — is harmless rather than a KeyError.
21
An empty set is falsy. After the discard, if no queues remain on this topic the branch fires. Subtle: reading self._subs[topic] here would itself re-create the topic via defaultdict if line 20 had deleted it — but line 20 only empties the set, never deletes the key, so the read is safe.
22
pop(topic, None) deletes the now-empty topic. The None default makes the delete idempotent. This is the only line that shrinks _subs — without it, every app ever watched would leave a permanent empty-set entry behind.

Why the cleanup matters

The /__events handler calls unsubscribe from a finally block — it runs the moment a browser tab disconnects. Without line 21–22, a long-running server that has served hundreds of short-lived tabs would accumulate hundreds of dead topic keys. Pruning empty topics keeps _subs proportional to currently open connections, not lifetime traffic.

05 — method three

publish  the fan-out.

The only async method, and the one the watcher calls. It drops the topic name into every queue currently subscribed to that topic.

pubsub.pylines 24 — 26
24    async def publish(self, topic: str) -> None:
25        for q in list(self._subs.get(topic, ())):
26            await q.put(topic)
25
.get(topic, ()) — not self._subs[topic]. The empty-tuple default looks the topic up without mutating the dict. A publish to a topic nobody is listening on must not create a topic entry that unsubscribe would then never get the chance to clean up.
25
Why list(...) wraps the set. The loop iterates a snapshot. The very next line awaits — yielding control back to the event loop — and a subscriber waking up may call unsubscribe, mutating _subs[topic]. Iterating the live set across an await would risk a "set changed size during iteration" RuntimeError. The copy makes the recipient list immune to that.
26
await q.put(topic) appends the topic name to each queue. The queues are unbounded, so put never actually blocks — but it is still awaited because Queue.put is a coroutine. Every subscribed tab now has a message waiting.
producer
Watcher sees a file change → await pubsub.publish("myapp")
routing
publish looks up _subs["myapp"] → a set of N queues
fan-out
the string "myapp" is put into all N queues
consumers
N waiting /__events handlers wake from q.get() → each writes an SSE change frame → N browser tabs reload

How this closes the live-reload loop

watcher.py publishes two kinds of topic: an app name when that app's files change, and INDEX_TOPIC ("__index") when the symlinks directory itself changes. server.py subscribes a tab to one of those topics depending on the ?app= query parameter. PubSub is the seam in the middle — the watcher and the HTTP layer never reference each other, only this broker. Swap the watcher out and PubSub would not notice.

06 — step back

Four things to take away.

The payload is the topic name

There is no event object, no JSON. A subscriber receiving "myapp" knows only "something under myapp changed" — which is exactly enough to trigger a location.reload(). Minimal protocol, minimal coupling.

defaultdict removes every existence check

subscribe can .add blind; unsubscribe can .discard blind. The one place auto-vivification would bite — publish — sidesteps it with .get(topic, ()).

Snapshot before you await

list(...) on line 25 is the one non-obvious line in the file. It is the discipline of never iterating a mutable collection across an await point in async code — concurrent mutation is otherwise a real possibility, not a theoretical one.

It cleans up after itself

_subs only ever holds topics with live subscribers. The broker has no background task, no expiry timer — just the discipline that unsubscribe prunes empty topics the instant they empty.
