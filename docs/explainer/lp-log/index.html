lp-log.js
sends bugs home.

When a user hits a glitch in the dashboard, the browser side of the failure is normally invisible — it lives in a console nobody is looking at. This script catches those front-end errors and forwards them to the server via POST /__log, so they land in the same log file as the backend's. One file to send, one place to look.

01 — the send path

One function ferries every report to the server.

The whole file is an IIFE, so it leaves no globals behind except the one it deliberately exposes. At its centre is send() — the single path every report takes to reach /__log. It is capped, beacon-first, and engineered never to throw.

lp-log.jslines 5 — 28
5(function () {
6  var ENDPOINT = "/__log";
7  var sent = 0;
8  var MAX = 25; // cap per page load so an error loop can't flood the log
9
10  function send(payload) {
11    if (sent >= MAX) return;
12    sent++;
13    try {
14      var body = JSON.stringify(payload);
15      if (navigator.sendBeacon) {
16        navigator.sendBeacon(ENDPOINT, new Blob([body], { type: "application/json" }));
17      } else {
18        fetch(ENDPOINT, {
19          method: "POST",
20          headers: { "Content-Type": "application/json" },
21          body: body,
22          keepalive: true,
23        });
24      }
25    } catch (_) {
26      // Logging must be invisible — swallow everything.
27    }
28  }
7 — 8
Two counters guard the page. sent tracks how many reports have gone out; MAX is the ceiling — 25 per page load. An error inside a render loop could otherwise fire thousands of times, drowning the log.
11 — 12
The cap is enforced first: once sent reaches MAX every further call is a no-op. The counter is incremented before the send so even a payload that fails to serialize still counts against the budget.
15 — 16
navigator.sendBeacon is preferred: it hands the request to the browser to deliver in the background, surviving even a page that is unloading. The payload is wrapped in a Blob tagged application/json.
17 — 24
Where sendBeacon is unavailable, it falls back to fetch with keepalive: true — the same "deliver even if the page goes away" guarantee, by a different route.
13 / 25 — 27
The entire body sits inside try { … } catch (_) {}. Logging must stay invisible: if the report itself can't be built or sent, the error is swallowed silently. send() can never throw into the page it is watching.
02 — two ways in

Automatic for uncaught, manual for caught.

Errors arrive at send() by two routes. The global hooks catch what no one handled; window.lpReport catches what someone already did.

34  window.lpReport = function (message, extra) {
35    var payload = {
36      message: String(message),
37      page: location.pathname,
38      userAgent: navigator.userAgent,
39    };
40    if (extra && extra.stack) payload.stack = String(extra.stack);
41    send(payload);
42  };

44  window.addEventListener("error", function (e) {
45    send({
46      message: String((e && e.message) || (e && e.type) || "error"),
47      source: (e && e.filename) || "",
48      line: (e && e.lineno) || "",
49      column: (e && e.colno) || "",
50      stack: e && e.error && e.error.stack ? String(e.error.stack) : "",
51      page: location.pathname,
52      userAgent: navigator.userAgent,
53    });
54  });
55
56  window.addEventListener("unhandledrejection", function (e) {
57    var r = e && e.reason;
58    send({
59      message: "unhandledrejection: " + (r && r.message ? r.message : String(r)),
60      stack: r && r.stack ? String(r.stack) : "",
61      page: location.pathname,
62      userAgent: navigator.userAgent,
63    });
64  });
44 — 54
The "error" hook catches any uncaught exception that bubbles to window. It records the message, the offending file (filename), the lineno / colno, and — when present — the real stack from e.error.stack.
56 — 64
The "unhandledrejection" hook catches the promise equivalent: a rejected promise no .catch() ever handled. It reads e.reason, preferring its .message, and prefixes the report so the two failure kinds stay distinguishable in the log.
34 — 42
window.lpReport(message, extra) is the manual path. Some errors are caught locally — a try/catch around a fetch, say — and so never reach the global "error" hook. This lets that code report deliberately. extra.stack is attached only when supplied.

Why a bare global, not window.LP?

lpReport is attached straight to window rather than hung off the window.LP namespace. The reason is load order: lp-log.js is the first script on the dashboard, loading before lp-util.js defines window.LP. If it tried to attach to LP the object wouldn't exist yet. Callers guard with window.lpReport && window.lpReport(…) — and delete.js uses exactly this to report a failed delete.

03 — scope

The dashboard only — never the served apps.

This is the deliberate boundary of the file, and it is easy to get wrong. Error capture runs on the localhost-pages dashboard and nowhere else.

lp-log.js is loaded as a plain <script> tag in templates/index.html — the dashboard's own page. It is placed first, before the other dashboard scripts, so it is already listening when any of them throws. Its job is to catch bugs in our front end.

It is deliberately not injected into served user apps. inject.py splices only reload.js into served pages and explicitly leaves error capture out. A served app is the user's own HTML; capturing its runtime errors would be noise — those are the user's bugs, not localhost-pages bugs — and injecting a logger would modify content that isn't ours to touch.

Contrast with reload.js

reload.js is injected into every served page, because live-reload is the core feature those pages exist to get. lp-log.js is the opposite case: it is infrastructure for the dashboard, so it stays on the dashboard. Two scripts, two scopes — one injected everywhere, one loaded in exactly one place.
