watcher.py
refuses to stay dead.

This module is the eyes of localhost-pages. It watches the symlinks directory and the resolved target of every link, and on any change publishes a topic name onto the PubSub — which the SSE stream turns into a browser reload. At 136 lines it is the largest module, and almost half of it is resilience: an OS file-watcher that glitches must never permanently blind the app.

02 — the contract

The class docstring is the design.

Before a line of logic, the docstring states both the job and the failure model. Read it carefully — the rest of the file is just that paragraph made executable.

watcher.pylines 16 — 26
16class Watcher:
17    """Watches the symlinks directory and the resolved target of each link.
18    On any change, publishes the affected app name (or INDEX_TOPIC) onto the
19    given PubSub. Debounces bursts of events.
20
21    Resilience: each watch coroutine wraps its awatch() loop in a retry
22    loop with exponential backoff so an unexpected exception (FSEvents
23    glitch, transient OSError) doesn't permanently kill the watch. A
24    periodic reaper task also re-syncs every REAP_INTERVAL_S seconds so
25    any task that did exit gets recreated even without an external
26    trigger."""

The job (lines 17–19): two kinds of thing are watched — the symlinks directory itself, and the resolved target of each individual link. A change to the directory means an app was registered or removed; a change inside a target means an app's own files were edited. Either way a topic name is published, and bursts are debounced so a save that touches ten files fires one event, not ten.

The failure model (lines 21–26): an OS file-watcher is not reliable infrastructure. FSEvents can glitch; a transient OSError can surface mid-stream. The watcher therefore assumes its own awatch() loops will die, and is built to notice and recover. Two mechanisms are named here — per-loop exponential-backoff retry, and a periodic reaper — and a third, the clean-exit path, is described in section 06.

Why this matters so much

If an awatch() loop dies silently, the symptom is invisible: the page simply stops live-reloading, with no error the user ever sees. There is nothing to alert on. The only defence is to make silent death structurally impossible — which is exactly what the next five sections build.

04 — lifecycle

start() & stop()

The two methods server.py's lifespan hook calls. start spins up the machine; stop tears every part of it down.

watcher.pylines 40 — 53
40    async def start(self) -> None:
41        self._tasks.append(asyncio.create_task(self._watch_symlinks_dir()))
42        self._tasks.append(asyncio.create_task(self._reaper()))
43        await self._sync_app_watches()
44
45    async def stop(self) -> None:
46        self._stop.set()
47        for t in self._tasks:
48            t.cancel()
49        for t in self._app_tasks.values():
50            t.cancel()
51        await asyncio.gather(*self._tasks, *self._app_tasks.values(), return_exceptions=True)
52        self._tasks.clear()
53        self._app_tasks.clear()
41 — 42
The two long-lived tasks are created and parked in _tasks: the symlinks-dir watcher and the reaper. create_task schedules them; it does not block.
43
_sync_app_watches() is awaited once here, so the per-app watches exist before start() returns — when the app begins serving requests, every already-registered app is already watched.
46
Step one of shutdown. Setting _stop asks every loop to exit gracefully at its next check — and, because the same event is passed as awatch's stop_event, the generators themselves end.
47 — 50
cancel() on top of the stop flag — belt and braces. A task blocked somewhere that does not check _stop still gets a CancelledError and unwinds.
51
gather(…, return_exceptions=True) waits for every task to actually finish. return_exceptions=True means a CancelledError or stray exception is collected rather than re-raised — shutdown never fails partway.
52 — 53
Both collections are cleared, leaving the Watcher in a clean, re-startable state.
05 — the resilience model

Three layers against silent death.

No single mechanism is trusted. A failure mode that slips past one layer is caught by the next — defence in depth, applied to an unreliable OS file-watcher.

1

Per-loop backoff retry

Each watch wraps awatch() in a while not _stop loop. An unexpected exception is logged, the loop sleeps with exponential backoff (1→30 s), then re-enters awatch(). A glitch is a hiccup, not a death.

2

FileNotFoundError clean exit

If a watched target vanishes, retrying is pointless — there is nothing to watch. That one exception is caught separately and the task returns cleanly, deliberately leaving its slot for the reaper or the dir-watcher to reconcile.

3

The periodic reaper

Every 60 s the reaper re-runs _sync_app_watches(). Any per-app task that exited — crash or clean return — is detected as done() and recreated. The safety net beneath the other two.

∞

The stop event ties it together

Every loop and every backoff sleep checks _stop. Resilience never fights shutdown: the same flag that survives crashes also guarantees a clean, prompt exit.

07 — the directory watcher

_watch_symlinks_dir()

The first long-lived task. It watches the symlinks directory itself — not recursively — so it sees links being added or removed, and re-derives the per-app watches whenever they do.

watcher.pylines 55 — 72
55    async def _watch_symlinks_dir(self) -> None:
56        backoff = self.BACKOFF_INITIAL_S
57        while not self._stop.is_set():
58            try:
59                async for _ in awatch(self.symlinks_dir, stop_event=self._stop,
60                                      debounce=self.debounce_ms, recursive=False):
61                    await self.pubsub.publish(INDEX_TOPIC)
62                    await self._sync_app_watches()
63                    backoff = self.BACKOFF_INITIAL_S
64                return  # awatch exited cleanly because _stop was set
65            except Exception:
66                logger.exception("symlinks dir watch crashed; restarting in %.1fs", backoff)
67                try:
68                    await asyncio.wait_for(self._stop.wait(), timeout=backoff)
69                    return  # asked to stop while sleeping
70                except asyncio.TimeoutError:
71                    pass
72                backoff = min(backoff * 2, self.BACKOFF_MAX_S)
57
The outer retry loop. It keeps re-entering awatch() as long as shutdown has not been requested — so a crash inside means another attempt, not the end.
59 — 60
recursive=False — only the directory's own entries matter here; changes inside a target are _watch_app's job. stop_event=self._stop lets shutdown end the generator from outside.
61 — 62
On any directory change: publish INDEX_TOPIC so open index tabs reload, then _sync_app_watches() so the per-app watch set matches the new reality. The loop variable is _ — the fact of a change is all that is used, never its detail.
63
A successful event resets backoff to its initial value. Health is proven by progress — so a glitch hours later starts its retry sequence fresh from 1 s.
64
Reaching here means awatch() ended without raising — which only happens because _stop was set. return exits the retry loop for good. The trailing comment marks this as the intended clean exit.
65 — 66
Any other exception is caught broadly and logged with a full traceback via logger.exception. The watch is not abandoned — control falls through to the backoff sleep.
68 — 71
The interruptible sleep. Rather than asyncio.sleep(backoff), it wait_fors the stop event with a timeout. If stop fires first the task returns immediately; if the timeout wins, a TimeoutError is caught and the loop retries. Shutdown is never made to wait out a 30 s backoff.
72
Backoff doubles, clamped at BACKOFF_MAX_S (30 s). The classic exponential-backoff curve: forgiving of one-offs, gentle on a persistently broken watch.

Why except Exception and not a precise type

An OS file-watcher can fail in ways the code cannot enumerate — FSEvents quirks, transient OSErrors, library-internal faults. The contract is "survive anything", so the catch is deliberately broad. Note it is Exception, not bare except: — CancelledError derives from BaseException and so still propagates, keeping stop()'s cancel() effective.

08 — the reconciler

_sync_app_watches()

The pivot of the whole module. It computes the set of apps that should be watched, diffs it against the tasks currently running, and makes reality match — cancelling the stale, spawning the missing.

watcher.pylines 74 — 89
74    async def _sync_app_watches(self) -> None:
75        apps = scan(self.symlinks_dir)
76        wanted = {a.name: a for a in apps if a.target is not None and a.error is None}
77
78        for name in list(self._app_tasks):
79            if name not in wanted:
80                self._app_tasks[name].cancel()
81                del self._app_tasks[name]
82
83        for name, app in wanted.items():
84            existing = self._app_tasks.get(name)
85            if existing and not existing.done():
86                continue
87            self._app_tasks[name] = asyncio.create_task(
88                self._watch_app(name, app.target)
89            )
75 — 76
scan() returns one App per symlink. wanted keeps only the healthy ones — a resolvable target and no error. Broken symlinks and missing-index.html apps are silently excluded; there is nothing watchable about them.
78 — 81
The removal pass. Iterating a list() copy of the keys (the dict is mutated inside the loop), any running task whose app is no longer wanted is cancelled and dropped. This is how a deleted symlink stops its watch.
83 — 86
The addition pass. For each wanted app, if a task already exists and is not done(), leave it — continue. A live healthy watch is never disturbed by a re-sync.
87 — 89
Otherwise — no task, or one that has finished — spawn a fresh _watch_app for it. This single line is what makes the method a true reconciler: it both adds genuinely new apps and resurrects any whose watch died.

One method, three callers

_sync_app_watches is called from start() (initial population), from _watch_symlinks_dir (event-driven, the moment a link changes), and from _reaper (time-driven, every 60 s). Because it is idempotent — it computes desired state and converges on it — calling it from three unrelated triggers is safe. A redundant call simply does nothing.

09 — the per-app watcher

_watch_app()

One instance of this runs per registered app. It watches that app's target directory recursively, and on any change publishes twice. It also carries the second resilience layer — the FileNotFoundError clean exit.

watcher.pylines 91 — 116
91    async def _watch_app(self, name: str, target: Path) -> None:
92        backoff = self.BACKOFF_INITIAL_S
93        while not self._stop.is_set():
94            try:
95                async for _ in awatch(target, stop_event=self._stop,
96                                      debounce=self.debounce_ms, recursive=True):
97                    await self.pubsub.publish(name)
98                    # Also reload the index so the per-row "NEW" indicator
99                    # picks up the freshness change.
100                    await self.pubsub.publish(INDEX_TOPIC)
101                    backoff = self.BACKOFF_INITIAL_S
102                return
103            except FileNotFoundError:
104                # Target went away (symlink target deleted). Don't retry —
105                # the symlinks-dir watcher will notice and call
106                # _sync_app_watches, which drops us from the dict.
107                logger.info("app %s target vanished; exiting watch", name)
108                return
109            except Exception:
110                logger.exception("app %s watch crashed; restarting in %.1fs", name, backoff)
111                try:
112                    await asyncio.wait_for(self._stop.wait(), timeout=backoff)
113                    return
114                except asyncio.TimeoutError:
115                    pass
116                backoff = min(backoff * 2, self.BACKOFF_MAX_S)
95 — 96
recursive=True — the opposite of the dir watcher. An app's files can sit in nested folders, and a save to any of them must trigger a reload, so the entire subtree is watched.
97
First publish: the app's own topic (its name). The SSE stream for any tab open on /{name}/… is subscribed to exactly this topic, and reloads.
98 — 100
Second publish: INDEX_TOPIC. Editing an app's files also changes its freshness — its index.html mtime moves — and the index page renders a per-row "NEW" indicator from that. Publishing the index topic too keeps that badge accurate without the user reloading the index by hand.
101 — 102
As in the dir watcher: a successful event resets backoff; a clean awatch() exit (only via _stop) returns the task for good.
103 — 108
The clean-exit special case. FileNotFoundError means the watched directory itself is gone — the symlink target was deleted. Retrying with backoff would be pointless: there is nothing left to watch. The task logs at info level (this is normal, not an error) and returns.
109 — 116
Every other exception takes the standard backoff-retry path, identical to the dir watcher's — a glitch is survived, a vanished target is not pointlessly chased.

Why a vanished target exits instead of retrying

The comment on lines 104–106 spells out the choreography. When a target disappears, the symlinks-dir watcher sees the directory change, runs _sync_app_watches, finds the app no longer in wanted, and removes it from _app_tasks. So this task returning is correct: it deliberately steps aside and lets the reconciler do the bookkeeping. Two coroutines, one consistent outcome — no orphaned task, no dict left out of sync. And even if the dir-watcher event were somehow missed, the reaper would catch the now-done() task within 60 s.

10 — the safety net

_reaper()

The second long-lived task and the third resilience layer. It assumes the event-driven machinery can fail, and re-synchronises on a fixed clock regardless.

watcher.pylines 118 — 136
118    async def _reaper(self) -> None:
119        """Periodically resync app watches. Cleans up any per-app task that
120        exited (crash or clean return) and recreates watches for newly
121        wanted apps that are missing. This is the safety net against silent
122        watcher death."""
123        while not self._stop.is_set():
124            try:
125                await asyncio.wait_for(self._stop.wait(), timeout=self.REAP_INTERVAL_S)
126                return
127            except asyncio.TimeoutError:
128                pass
129            dead = [n for n, t in self._app_tasks.items() if t.done()]
130            for n in dead:
131                logger.warning("reaper: app watch %s was dead, recreating", n)
132                del self._app_tasks[n]
133            try:
134                await self._sync_app_watches()
135            except Exception:
136                logger.exception("reaper: _sync_app_watches failed")
125 — 128
The same interruptible-sleep idiom as the backoff path, used here as a plain 60 s tick. wait_for on the stop event: if stop fires the reaper returns at once; otherwise the TimeoutError is the signal that an interval elapsed and there is work to do.
129 — 132
The reap. Every done() per-app task — finished by crash or by clean return — is collected and deleted from the dict. The warning log makes a watch that needed resurrecting visible in the logs.
133 — 134
Then _sync_app_watches(). With the dead entries already removed, its addition pass sees those apps as missing and respawns their watches — and it also picks up any app added since the last tick. One call, both jobs.
135 — 136
Even the reaper's own re-sync is wrapped: if _sync_app_watches throws, it is logged and swallowed so the while loop survives to the next tick. The safety net itself cannot fall.

Why a reaper at all, when there's already backoff retry?

Backoff retry only saves a loop that crashes with a catchable exception. It cannot save a task that the event loop kills outright, or one that returns cleanly (the FileNotFoundError path) and leaves a stale slot, or — more subtly — a symlinks-dir watcher event that was simply missed, so a new app never got a watch. The reaper closes all of those gaps by re-deriving desired state from the filesystem on a fixed clock. It does not need to understand how a watch went missing; it only needs to notice that it did. That is what makes it a true safety net — bounded recovery time, no matter the cause.

11 — step back

Four things to take away.

The module is built around its own unreliability

An OS file-watcher will glitch — that is taken as given. Every loop assumes it will die, every backoff sleep checks for shutdown, and a periodic reaper re-derives the truth from disk. Resilience here is not a feature bolted on; it is the shape of the file.

Idempotent reconciliation beats event tracking

_sync_app_watches never trusts a stream of "app added / app removed" events. It re-computes the full desired set and converges on it. That is why three unrelated callers — startup, the dir watcher, the reaper — can all invoke it safely and redundantly.

One stop event, threaded everywhere

A single asyncio.Event is the loop condition, the awatch stop_event, and the wait_for target inside every sleep. Shutdown is therefore both prompt and total — no loop and no backoff timer can outlive stop().

Every file change publishes a topic — that is the whole external effect

The watcher writes nothing and serves nothing. Its entire output is pubsub.publish(…): an app's own name, and/or INDEX_TOPIC. server.py's SSE route turns those topic names into change events, and the injected reload.js turns those into a browser reload. This module is one link in that chain.
