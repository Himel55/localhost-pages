keys.js
hands on the home row.

Seven key bindings, zero state. keys.js moves a .kbd-focus highlight around the list — then delegates every real action by calling .click() on whichever button the focused row already owns. It never reimplements delete, pin, or edit. It just presses the button.

01 — the shortcut map

Seven bindings. Every one declared upfront.

The file opens with an exact manifest of what it handles — and what it intentionally leaves to others. / and Esc are called out explicitly because those are shared keys that other modules touch too.

j / ArrowDown
Focus the next visible row — skips rows hidden by search filtering.
k / ArrowUp
Focus the previous visible row.
Enter
Open the focused row — delegates to row.click(), which peek.js intercepts to open the preview modal.
x
Arm or confirm delete — finds the row's .del button and calls .click(). Logic lives entirely in delete.js.
p
Toggle pin on the focused row — delegates to the row's .pin button. Logic lives in pin.js.
e
Toggle edit mode — delegates to .edit-toggle. Logic lives in edit.js.
Esc
Clear keyboard focus — but only when no other module owns Esc. Described in section 04.

/ is deliberately absent from the switch

The header comment mentions / for search — but the binding itself is registered in search.js. keys.js documents it here only so the full keyboard contract is readable in one place.

02 — focus tracking

Three helpers. One class, moved around.

The entire focus model is a single CSS class, .kbd-focus, on at most one row at a time. Three functions manage it — and a fourth moves the focus by a delta.

keys.jslines 15 — 38
15  function visibleRows() {
16    return Array.from(list.querySelectorAll('.row')).filter((r) => r.style.display !== 'none');
17  }
18
19  function focusedRow() {
20    return document.activeElement?.closest?.('.row') || document.querySelector('.row.kbd-focus');
21  }
22
23  function setFocus(row) {
24    list.querySelectorAll('.row.kbd-focus').forEach((r) => r.classList.remove('kbd-focus'));
25    if (!row) return;
26    row.classList.add('kbd-focus');
27    row.scrollIntoView({ block: 'nearest' });
28  }
29
30  function move(delta) {
31    const rows = visibleRows();
32    if (!rows.length) return;
33    const current = focusedRow();
34    let i = rows.indexOf(current);
35    if (i === -1) i = delta > 0 ? -1 : rows.length;
36    const next = rows[Math.max(0, Math.min(rows.length - 1, i + delta))];
37    setFocus(next);
38  }
15 — 17
visibleRows() filters by display !== 'none' — the exact test search.js uses to hide non-matching rows. Navigation automatically skips filtered-out rows without any coordination.
19 — 21
focusedRow() checks activeElement first — if a button inside a row has real DOM focus, that row is the focused one. The .kbd-focus class is the fallback for when nothing has DOM focus.
23 — 28
setFocus() removes .kbd-focus from every row before adding it to the new one — enforcing the invariant that at most one row is highlighted. Passing null clears focus entirely. scrollIntoView with nearest avoids jarring jumps.
34 — 35
move(delta) handles the "nothing focused yet" case cleanly. When indexOf returns -1, pressing j (delta > 0) snaps to index 0; pressing k (delta < 0) snaps to the last row — whichever end is natural for the direction of travel.
03 — the keydown switch

No logic. Just button clicks.

The keydown handler first filters out the two contexts where bindings must never fire — then dispatches. The key insight: every case that touches a row finds the appropriate button and calls .click().

keys.jslines 40 — 44
40  document.addEventListener('keydown', (e) => {
41    const tag = e.target.tagName;
42    if (tag === 'INPUT' || tag === 'TEXTAREA' || tag === 'SELECT') return;
43    if (e.metaKey || e.ctrlKey || e.altKey) return;
42
All bindings are no-ops when the user is typing. Checking tagName is the lightest possible guard — no class inspection, no attribute lookup.
43
Modifier keys are ignored wholesale. Cmd+j, Ctrl+k, and similar chords are left to the browser or OS.
keys.jslines 56 — 93
56    switch (e.key) {

58      case 'ArrowDown':

75      case 'x': {

77        if (!row) return;

79        row.querySelector('.del')?.click();
80        return;
81      }
82      case 'p': {
83        const row = focusedRow();
84        if (!row) return;
85        e.preventDefault();
86        row.querySelector('.pin')?.click();
87        return;
88      }
89      case 'e':
90        e.preventDefault();
91        document.querySelector('.edit-toggle')?.click();
92        return;
93    }
71
Enter calls row.click() on the whole row element — not on any button. peek.js has its own click listener on .row elements; this key triggers that handler without keys.js knowing anything about the preview.
79
x finds .del inside the row and clicks it. The two-step arm/confirm dance, the DELETE fetch, the timeout reset — none of that is here. It all lives in delete.js.
86
p finds .pin and clicks it. The PATCH request, optimistic UI update, and error rollback are in pin.js.
91
e is the one binding that doesn't need a focused row — edit mode is global. It locates .edit-toggle (a single element in the page header) and delegates to edit.js.

The optional-chaining ?. is the entire guard

If a row has no .del button — perhaps because edit mode is off and the button is not rendered — the optional chain short-circuits silently. No conditionals, no feature flags. The key is a no-op whenever the affordance is absent.

04 — cooperative Esc

Esc only fires when nothing else needs it.

Escape is shared across several modules. keys.js checks a busy flag before acting — standing down when any other handler has prior claim.

keys.jslines 48 — 54
48    if (e.key === 'Escape') {
49      const busy = document.querySelector('.peek-backdrop.open, .help-backdrop.open')
50        || document.body.classList.contains('editing')
51        || !document.querySelector('.sort-menu[hidden]');
52      if (!busy) setFocus(null);
53      return;
54    }
49
.peek-backdrop.open — the preview modal is open. .help-backdrop.open — the help overlay is showing. Either truthy element means Esc belongs to those handlers, not here.
50
body.editing — edit mode is active. edit.js owns Esc in that context to exit edit mode.
51
!querySelector('.sort-menu[hidden]') — the sort menu is open when the element exists but lacks the hidden attribute. sort.js closes it on Esc.
52
Only when none of the above is true does keys.js act — calling setFocus(null) to clear the keyboard highlight.

The layered-Esc convention, across all modules

There is no central Esc dispatcher. Instead, each module reads DOM state to decide whether Esc is "theirs." The busy check in keys.js is the lowest-priority claim — it acts only when every higher-priority handler has already cleaned up. This pattern lets any module own Esc during its active state without coordinating through shared code.
