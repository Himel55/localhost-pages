localhost-pages/technical-explanation/__main__.py live
One file, read closely

__main__.py
is the front door.

Run python -m localhost_pages and this is the first line that executes. It is 33 lines: it switches on always-on logging (via logconfig.py), reads four environment variables, logs a startup banner, asks server.py for a built app, and hands that app to uvicorn. It owns no logic of its own — it is pure wiring.

01 — top of the file

Six imports, two of them siblings.

The imports tell you exactly what an entrypoint needs: a way to read the environment, a way to touch the filesystem, an ASGI server, and two siblings — the logging setup and the app itself. A module-level logger is created right after.

Why create_app is imported, not app

server.py deliberately exposes a factory function rather than a module-level app object. That lets this entrypoint decide the symlinks_dir at runtime — and lets the test suite build isolated apps pointed at throwaway directories. The cost here is one extra call on line 26.

02 — the only function

def main()  the whole program.

Everything below lives inside one function. It takes no arguments and returns None — its entire job is side effects, in order.

__main__.pyline 15
15def main() -> None:

The -> None annotation is honest: main() produces no value. It runs four steps — configure logging, read the environment, build the app, run the server — and the fourth step blocks forever. Naming it main rather than running code at module scope keeps the file importable without launching a server.

03 — step one

Logging: always on, in one call.

Before anything else, main() turns logging on. There is no branch and no format string here anymore — it is a single call into logconfig.py.

__main__.pylines 16 — 18
16    # Logging is always on (rotating file + stderr). LOCALHOST_PAGES_LOG still
17    # overrides the file location for users who want logs somewhere specific.
18    setup_logging(os.environ.get("LOCALHOST_PAGES_LOG"))
18
One call, always on. setup_logging(...) configures both a rotating log file and stderr — every run is logged, with no flag to remember and no opt-in. Logs always land somewhere discoverable.
18
The env var only overrides the path. os.environ.get("LOCALHOST_PAGES_LOG") returns a custom file location or None. Unset, setup_logging falls back to a platform-appropriate path; set, it writes there instead. It no longer flips between two modes — the file always exists, this just decides where.

The real work lives in logconfig.py

The rotating-file handler, the platform log path, directory creation, the format string, and the startup banner all moved out of this file into the sibling logconfig.py. __main__.py just calls setup_logging and trusts it. For the rotation policy and banner internals, read logconfig.py.

04 — step two

Configuration is four environment variables.

No flags, no config file, no parser. The fourth variable — LOCALHOST_PAGES_LOG — was already consumed by the setup_logging call above; here three more lines read the symlinks directory, the host, and the port, each with a sensible built-in default.

__main__.pylines 20 — 22
20    symlinks = Path(os.environ.get("LOCALHOST_PAGES_SYMLINKS", DEFAULT_SYMLINKS))
21    host = os.environ.get("LOCALHOST_PAGES_HOST", "127.0.0.1")
22    port = int(os.environ.get("LOCALHOST_PAGES_PORT", "8080"))
20
Here get is given a second argument — the default returned when the variable is unset. The result is wrapped in Path(...) so it is a real path object whether it came from the environment (a string) or from DEFAULT_SYMLINKS (already a Path).
21
The bind address defaults to 127.0.0.1 — loopback only. The server is not reachable from other machines unless you deliberately override this with 0.0.0.0. A safe default for a tool literally named localhost-pages.
22
Environment variables are always strings, so the port is wrapped in int(...). The default "8080" is passed as a string too, so both paths feed int() identically. A non-numeric value here would raise ValueError on startup — failing loudly rather than silently.

Why config is environment-variables-only

There is no argparse, no YAML, no dotfile. Every knob is an environment variable with a default — which means the install scripts that launch this as a background service (a launchd plist, a systemd unit, a shell wrapper) can configure it entirely through the process environment, with no command line to construct and no file to ship. Unset everything and it still runs correctly. Configuration is the four os.environ.get calls — there is nothing else to learn.

05 — steps three & four

Banner, build the app, then hand it to uvicorn.

The tail of main(): it logs a startup banner, constructs the app, then starts the server and never returns.

__main__.pylines 24 — 29
24    log_startup_banner(logger, host=host, port=port, symlinks_dir=symlinks)
25
26    app = create_app(symlinks_dir=symlinks)
27    # access_log=False: our request-logging middleware already records every
28    # request, so uvicorn's access log would just duplicate it.
29    uvicorn.run(app, host=host, port=port, log_level="info", access_log=False)
24
The startup banner. log_startup_banner (from logconfig.py) writes the version, Python build, OS, and resolved config — host, port, symlinks directory — into the log at boot. When something goes wrong later, the very first lines of the log already say exactly what was running and how it was configured. Triage starts here.
26
The factory call. The symlinks path resolved on line 20 is passed by keyword into create_app, which builds and returns a fully configured FastAPI instance — its watcher, pubsub, static mount, and seven routes all wired up. This entrypoint never touches any of that; it only supplies the one parameter.
27 — 28
Why access_log=False. The request-logging middleware in server.py already records every request. Leaving uvicorn's own access log on would log each request twice, so it is switched off — the middleware is the single source of truth for request logs.
29
uvicorn.run(...) binds the socket and enters the event loop. The app, host, and port all flow in from the lines above. This call blocks — it runs until the process is interrupted, so it is the last statement that ever executes in main().

The lifespan handshake

Passing the app object to uvicorn.run is what eventually triggers the lifespan hook inside create_app — which creates the symlinks directory if missing and starts the file watcher. So this single line is also, indirectly, what turns on live reload. __main__.py sets the stage; server.py performs.

06 — the tail

The __main__ guard.

The last two lines decide whether main() actually runs — and the answer depends entirely on how Python loaded this file.

__main__.pylines 32 — 33
32if __name__ == "__main__":
33    main()

Python sets the module-level variable __name__ to the string "__main__" only when a file is executed directly as a program. If the same file is imported by another module, __name__ is the module's import path instead — and this branch is skipped.

The effect: main() runs when you invoke python -m localhost_pages, but the file stays safely importable. A test, or another module, can pull symbols from it without accidentally starting a server.

Why this file is named __main__.py

Inside a package, a file named __main__.py is the module Python runs when the package itself is the target of python -m. So python -m localhost_pages executes this file. The filename and the if __name__ == "__main__" guard are two halves of the same convention — together they make the package directly runnable.

07 — step back

Four things to take away.

An entrypoint should be wiring, not logic

All 33 lines do is read configuration and connect parts that live elsewhere. There is no HTTP handling, no file scanning, no business rule here — __main__.py is deliberately thin, and that is what makes it easy to read in one sitting.

Every setting has a default

Run python -m localhost_pages with an empty environment and it still works: always-on logging at the default path, the default symlinks directory, loopback, port 8080. The environment variables are overrides, never requirements.

The order of the four steps matters

Logging is configured first so that anything later — including create_app and the watcher it starts — can emit log lines that are actually captured. Configuration is read before the app is built; the app is built before the server runs. Each step depends on the one before it.

The factory boundary is shared with server.py

create_app and DEFAULT_SYMLINKS are imported rather than redefined, so the default and the construction logic exist in exactly one place. This file decides what to pass; server.py decides how to build. Read server.py next to see the other side of that line.
